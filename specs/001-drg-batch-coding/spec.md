# Feature Specification: DRG 批次編碼(Tw-DRG Batch Coding)

**Feature Branch**: `001-drg-batch-coding`
**Created**: 2026-06-11
**Status**: Draft
**Input**: User description: "DRG 批次編碼 的功能,反向寫成規格"

> 本規格由既有 legacy 編碼服務(反編譯之 `DRGICD10` / `rddi_lib`)的觀察行為**反向重建**,描述「做什麼/為何」,作為新世代系統的需求基準。對齊 legacy 行為屬憲章原則 II,但凡與官方標準衝突以標準為準(原則 I)。

## Clarifications

### Session 2026-06-11

- **FR-014(輸入/輸出格式)**: 輸入沿用 legacy 固定欄位格式;輸出改 **Unicode(UTF-8)** 取代 BIG5;欄位 schema 後續可調整。DRG/MDC/CC 數值仍須與 legacy 一致(BIG5→Unicode 為可接受偏移)。
- **FR-015(目標版次)**: 對應 **Tw-DRG 115/01/01**、應用版本 **v3.4.20**。
- **FR-016(批次錯誤語意)**: **逐筆獨立(single)** 續跑,一筆錯誤不影響他筆;輸出**同時含(both)** 成功與錯誤兩類;非 all-or-nothing。
- **FR-017(交付形態)**: **服務 / API / CLI**(headless),不含桌面 GUI。
- **SC-004(規模)**: 單批最大筆數**預設 10**,可由**環境變數**調整。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 將合法病歷檔批次編出 DRG 結果檔 (Priority: P1)

醫療機構的申報人員提供一份住院病歷申報檔(每列一筆病患住院案件,含基本資料、診斷碼、手術碼、日期、轉歸與醫療費用)。系統逐筆套用 Tw-DRG 分組規則,為每筆案件指派 DRG、MDC 與併發症/合併症(CC)分級,並輸出一份結果檔供申報與分析。

**Why this priority**: 這是整個功能存在的理由——把病歷轉成可申報的 DRG。沒有它其餘皆無意義;單獨完成即為可用 MVP。

**Independent Test**: 餵入一份**全部合法**的範例病歷檔,確認輸出結果檔中每筆案件都有 DRG、MDC、CC 欄位,且與官方 Tw-DRG 參考結果相符。

**Acceptance Scenarios**:

1. **Given** 一份所有欄位皆合法的病歷檔,**When** 執行批次編碼,**Then** 產出結果檔,且每筆案件含 DRG 碼、MDC、CC 分級與(空的)錯誤註記。
2. **Given** 一筆主診斷對應某 MDC 且有對應手術的案件,**When** 編碼,**Then** 指派之 DRG/MDC 與官方分組結果一致。
3. **Given** 一筆主診斷屬精神科範疇(對應特殊類別)的案件,**When** 編碼,**Then** 回傳對應之特殊/哨兵分組結果而非一般 DRG。
4. **Given** 輸入檔的筆數為 N,**When** 編碼完成,**Then** 結果檔的案件筆數 = N(無任何案件被靜默丟棄)。

---

### User Story 2 - 輸入驗證與錯誤標註(不中斷整批) (Priority: P2)

申報檔常含格式錯誤(日期空白/格式不符、主診斷缺漏、碼長度超限、出院日早於版本生效日)。系統須逐筆驗證,將不合格案件以**人類可讀的中文原因**標註於結果中,且不因單筆錯誤中止整批。

**Why this priority**: 真實資料必有髒資料;若一筆壞資料即中止全批,功能在現場不可用。但需先有 US1 的編碼主幹才有意義,故 P2。

**Independent Test**: 餵入一份**混合合法與多種錯誤**的病歷檔,確認合法筆數正常編碼、錯誤筆數帶明確中文錯誤原因、且整批仍跑完輸出。

**Acceptance Scenarios**:

1. **Given** 某筆缺主診斷碼,**When** 編碼,**Then** 該筆標註「主診斷碼格式錯誤」類原因,其餘筆不受影響。
2. **Given** 某筆出院日為空或格式不符,**When** 編碼,**Then** 該筆標註對應日期錯誤原因。
3. **Given** 某筆診斷碼或手術碼長度超出上限,**When** 編碼,**Then** 該筆標註碼長度錯誤,並回報出問題的列號。
4. **Given** 某筆出院日早於本版生效日,**When** 編碼,**Then** 該筆標註版本限制日期錯誤。
5. **Given** 一筆 CSV 結構毀損(欄位數不符),**When** 編碼,**Then** 系統略過該列並記錄,不致整批崩潰。

---

### User Story 3 - 無人值守 / 自動化執行 (Priority: P3)

系統可由排程或上游系統以非互動方式啟動,指定輸入與輸出,完成後自動結束,無需人工點選,以便整合進申報作業流程。

**Why this priority**: 提升維運效率與整合性,但非首要;互動式手動執行已能交付價值,故 P3。

**Independent Test**: 以自動模式啟動一次編碼(不經人工互動),確認其讀檔、編碼、輸出結果檔後自行結束,且結果與互動模式一致。

**Acceptance Scenarios**:

1. **Given** 指定了輸入與輸出位置的自動模式,**When** 啟動,**Then** 全程無互動完成並產出結果檔。
2. **Given** 自動模式執行中,**When** 完成,**Then** 程序自動結束並可由結果檔/狀態判定成功與否。

---

### Edge Cases

- CSV 某列欄位數不符(MalformedLine)→ 略過並記錄,不中止整批。
- 主診斷碼缺漏或為空 → 標註錯誤,不送入分組。
- 入院日 / 出生日為 `0000/00/00` 或格式不符 → 依規則改用申報起日或子女出生日計齡;出院日無效則標錯。
- 新生兒就附母親案件(part code 特例)且有子女出生日 → 以子女出生日計齡。
- 性別欄空白 → 由身分證字號推導(無法判定則記為未知)。
- 案件無法分群 → 回傳哨兵/錯誤分組碼並標示,而非靜默無值。
- 診斷碼或手術碼長度超出上限 → 標註並回報列號。
- 出院日早於本版生效日 → 標註版本限制。
- 超大輸入檔 → 須能完成而不耗盡資源(規模上限見 SC / 待釐清)。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 接收一份批次病歷申報檔,每列代表一筆住院案件,含機構代號、費用年月、身分識別、性別、入院/出生/出院日期、轉歸代碼、醫療費用、案件類別,及多筆(主+次)診斷碼與多筆(主+次)手術碼。
- **FR-002**: 系統 MUST 逐筆驗證:入院日與出生日存在且格式正確、出院日存在且格式正確、主診斷碼存在、各診斷/手術碼長度未超上限、出院日不早於本版生效日。
- **FR-003**: 系統 MUST 為每筆合法案件,依**指定版次的官方 Tw-DRG 分組規則集**指派 DRG 碼、MDC、CC 分級與錯誤碼。
- **FR-004**: 系統 MUST 計算病患年齡(歲/月/日),並於新生兒就附母親案件時改以子女出生日為基準。
- **FR-005**: 系統 MUST 由次診斷判定併發症/合併症(CC/MCC)分級,且**排除臨床上已被主診斷涵蓋**的次診斷(不重複計入)。
- **FR-006**: 系統 MUST 將不合格案件以人類可讀原因(中文)標註,且**不得因單筆錯誤中止整批**;結構毀損的列略過並記錄。
- **FR-007**: 系統 MUST 確保輸入檔每一筆案件都出現在輸出(合法者已編碼、不合法者帶錯誤註記),**不得靜默丟棄**。
- **FR-008**: 系統 MUST 產出結果輸出,內容含原始案件欄位加上 DRG、MDC、CC 分級、錯誤碼/錯誤原因,並以**在地化(中文)欄位標題**呈現。
- **FR-009**: 系統 MUST 支援無人值守/自動化執行,不需人工互動即可完成一次批次。
- **FR-010**: 系統 MUST 在讀檔 → 編碼 → 產製結果三階段對外回報進度與狀態。
- **FR-011**: 系統 MUST 隔離不同批次/殘留資料(每次執行的資料互不干擾)。
- **FR-012**: 系統 MUST 記錄批次的開始與結束時間,供稽核與效能追蹤。
- **FR-013**: 系統 MUST 標註所採用的 Tw-DRG 規則集版次,使每筆結果可追溯其依據(憲章原則 I)。
- **FR-014**: 系統 MUST 沿用 legacy 的**輸入**格式(固定欄位、逗號分隔、每列一案件)以確保上游相容;**輸出**改採 **Unicode(UTF-8)** 編碼取代 legacy 之 BIG5,欄位 schema 可於後續版本調整。輸出之 DRG/MDC/CC **數值仍須與 legacy 逐筆一致**(原則 II);編碼由 BIG5→Unicode 屬已記錄之可接受偏移(documented divergence)。
- **FR-015**: 系統 MUST 對應 **Tw-DRG 115/01/01 版**規則集(對齊 legacy 應用版本 **v3.4.20** 及其 ICD-10 公告版次);所採版次須標註於每批結果(見 FR-013)。
- **FR-016**: 系統 MUST 以**逐筆獨立(single-record)**方式處理:任一筆錯誤不影響其他筆;單批輸出**同時包含(both)**成功編碼與錯誤標註兩類案件。不採整批 all-or-nothing 中止。
- **FR-017**: 系統交付形態為**服務 / API / CLI(headless)**,不含桌面 GUI;US3 無人值守經 CLI/API 啟動,FR-010 之進度/狀態改以 API 回應或 CLI 輸出/日誌呈現。

### Key Entities *(include if feature involves data)*

- **病歷申報案件(Claim Encounter)**:一筆住院案件。屬性含機構代號、費用年月、身分識別、序號、性別、入院日、出生日、出院日、子女出生日、案件類別、轉歸代碼、醫療費用;關聯多筆診斷碼(1 主 + 多次)與多筆手術碼(1 主 + 多次)。
- **編碼結果(Coding Result)**:對應一筆案件的分組產出。屬性含 DRG 碼、MDC、CC 分級標記、CC 位元圖(哪些次診斷貢獻併發症)、錯誤碼、錯誤原因描述。
- **分組規則集(Grouping Ruleset)**:官方 Tw-DRG 標準的抽象表述——主診斷→MDC 對照、DRG 權重/決策樹、交叉/排除 ICD 規則、併發症(ECC)規則,具版次。
- **批次作業(Batch Job)**:一次執行。屬性含輸入位置、輸出位置、開始時間、結束時間、執行識別(用於資料隔離)。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 對任一輸入檔,輸出案件數 = 輸入案件數,合法者皆有 DRG/MDC/CC、不合法者皆有錯誤原因——**零靜默丟棄**(100%)。
- **SC-002**: 對官方公告的 Tw-DRG 測試案例集,系統輸出之 DRG/MDC/CC **逐筆與官方參考結果相符**(憲章原則 I/II 的驗收門檻)。
- **SC-003**: 不合格案件 100% 帶可定位、可行動的中文錯誤原因,維運人員可據以修正並重跑。
- **SC-004**: 單批處理筆數預設上限為 **10 筆**,且可由**環境變數**調整;在上限內單批須穩定完成而不耗盡資源。

## Assumptions

- 輸入為單一檔案、每列一筆案件、欄位以分隔符切分;主診斷位於固定欄位、次診斷與手術碼各有上限筆數(沿用 legacy 觀察:診斷與手術各上限約 20 筆)。
- 官方 Tw-DRG 分組規則集(對照表/權重/排除/ECC)以受控的參考資料形式提供且具版次,系統以該版為準。
- 病患性別缺漏時可由身分識別推導其值。
- 「對齊 legacy 行為」以可取得的 legacy 輸出樣本為比對基準;若無樣本,則以官方測試案例為準(原則 I 優先)。
- 申報用途的在地化(中文欄位標題、本國編碼慣例)為必要,但實際編碼/格式細節待 FR-014 釐清。
